#

Airport API — это Web API на ASP.NET Core для управления аэропортной инфраструктурой: рейсами, пассажирами, покупкой билетов и регистрацией на посадку. Приложение предназначено для демонстрации практики построения REST API с авторизацией, бизнес-логикой и контейнеризацией.

Сервис покрывает основные сценарии работы аэропорта: просмотр рейсов, создание профиля пассажира, оформление билета с расчётом цены по классу обслуживания и привязка пассажира к месту на рейсе. В текущей реализации логика регистрации и покупки билетов реализована в сервисном слое и использует данные из PostgreSQL.

#



- **Цена билета зависит от класса обслуживания:**
  - `Economy` — базовый тариф;
  - `Business` — умножение на `3`;
  - `First` — умножение на `5`.
- **Запрет на выбор чужого класса места:** при регистрации пассажиру запрещено выбирать кресло, которое не соответствует купленному классу обслуживания.
- **VIP-вытеснение:** VIP-пассажир может занять уже занятое место у обычного пассажира, при этом прежняя регистрация снимается и место становится доступным для пересмотра.
- **Автоназначение места:** если место не указано, система подбирает первое свободное место из схемы самолета.



- **RBAC:** в системе используются роли `Admin` и `User` для контроля доступа к защищённым операциям.
- **JWT-авторизация:** вход выполняется через `/api/auth/login`, после чего выдаётся JWT, который используется для доступа к защищённым маршрутам.
- **Хэширование паролей:** пароли пользователей хэшируются через `PasswordHasher<User>` перед сохранением в БД.
- **Публичная регистрация:** новый пользователь создаётся с ролью `User` без возможности выбрать произвольную роль из запроса.

#



Из корня проекта выполните:

```bash
docker-compose up -d --build
```

Команда поднимет два контейнера:

- `airport-postgres` — PostgreSQL;
- `airport-api` — ASP.NET Core API.



После успешного запуска API будет доступен по адресу:

- Swagger UI: `http://localhost:5254/swagger`



```bash
docker-compose down
```

#

При первом запуске в режиме `Development` приложение автоматически:

1. создаёт схему базы данных через `EnsureCreatedAsync()`;
2. заполняет тестовыми данными:
   - **самолёты** из `app/data/aircrafts.json`;
   - **аэропорты** и **гейты**;
   - **5 рейсов**;
   - **10 пассажиров**;
   - **супер-администратора** с логином `danya_admin` и паролем `airport_2026`.

> В текущей реализации данные инициализируются автоматически только при запуске в окружении `Development`, что соответствует настройке `ASPNETCORE_ENVIRONMENT=Development` в `docker-compose.yml`.

#



- `POST /api/auth/register` — регистрация нового пользователя (`User`);
- `POST /api/auth/login` — вход, получение JWT.



- `GET /api/flights` — получить список всех рейсов (табло);
- `GET /api/flights/{id}` — получить детали рейса;
- `GET /api/flights/{id}/passengers` — получить список пассажиров на рейсе (защищённый маршрут).



- `GET /api/passengers` — получить список пассажиров (защищённый маршрут);
- `POST /api/passengers` — зарегистрировать нового пассажира в системе.



- `POST /api/tickets/buy` — купить билет на рейс;
- `POST /api/tickets/checkin` — пройти регистрацию и назначить место;
- `DELETE /api/tickets/checkin/{ticketId}` — отменить регистрацию (защищённый маршрут).

#

В текущем проекте используются следующие слои:

- **model** — доменные сущности (`Flight`, `Passenger`, `Ticket`, `User`, `Aircraft`);
- **database** — `AirportDbContext` и конфигурация доступа к PostgreSQL;
- **dto** — DTO запросов и ответов;
- **services** — бизнес-логика (`CheckInService`, `FlightService`, `PassengerService`);
- **api** — группировка и регистрация HTTP-эндпоинтов.

#

- Авторизация выполняется через JWT, а не через cookie-сессию.
- В логике покупки билета цена рассчитывается сразу при создании билета и сохраняется в поле `PaidPrice`.
- Ресурс `app/data/aircrafts.json` используется как источник сидирования воздушных судов.
- В `docker-compose.yml` внешний порт `5254` пробрасывается в контейнерный порт `8080`.
